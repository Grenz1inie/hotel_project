## 问题一：SQL语法错误导致数据初始化失败

### 问题描述

在初始化订单数据时，执行`schema.sql`报SQL语法错误：

```sql
-- 错误的写法
INSERT INTO bookings (...) VALUES (...);  -- 这里用了分号
    (SELECT id FROM room_type ...),       -- 导致这行无法识别
```

**错误信息**：
```
You have an error in your SQL syntax; check the manual that corresponds 
to your MySQL server version for the right syntax near '(SELECT id FROM room_type'
```

### 问题分析

- SQL的`INSERT`语句使用多行`VALUES`插入数据时，应该用**逗号**分隔，而非分号
- 分号是语句结束符，导致后续的VALUES被当作新语句解析，引发语法错误

### 解决方案

修改`schema.sql`，将分号改为逗号：

```sql
-- 正确的写法
INSERT INTO bookings (hotel_id, room_type_id, room_id, user_id, ...)
VALUES
    (1, (SELECT id FROM room_type WHERE ...), ...),  -- 逗号
    (2, (SELECT id FROM room_type WHERE ...), ...),  -- 逗号
    (3, (SELECT id FROM room_type WHERE ...), ...);  -- 最后一行才用分号
```

### 运行结果

✅ **问题已解决**


## 问题二：空置率实时计算性能瓶颈

### 问题描述

管理后台每次查询空置率时，需要：
1. 遍历所有订单，统计已入住/已预订数量
2. 查询房型总数
3. 实时计算空置率 = (总数 - 已占用 - 已预订) / 总数

**性能表现**：
- 单次查询耗时：**3-5秒**
- 管理员频繁刷新时，数据库CPU飙升至80%+
- 用户体验差，图表加载缓慢

### 问题分析

根本原因是**架构设计问题**：
- 空置率数据属于**统计聚合结果**，不应每次实时计算
- 历史数据不会变化，重复计算浪费资源
- 缺少缓存机制，每次都全表扫描

### 解决方案

#### 方案一：数据库预计算架构

**1. 设计`vacancy_statistics`表**

```sql
CREATE TABLE vacancy_statistics (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    hotel_id BIGINT NOT NULL,
    room_type_id BIGINT NOT NULL,
    stat_date DATE NOT NULL COMMENT '统计日期',
    stat_hour TINYINT COMMENT '统计小时(0-23，NULL表示全天)',
    
    -- 统计指标
    total_rooms INT NOT NULL DEFAULT 0,
    available_rooms INT NOT NULL DEFAULT 0,
    occupied_rooms INT NOT NULL DEFAULT 0,
    reserved_rooms INT NOT NULL DEFAULT 0,
    
    -- 核心比率
    vacancy_rate DECIMAL(5, 4) NOT NULL DEFAULT 0.0000,
    occupancy_rate DECIMAL(5, 4) NOT NULL DEFAULT 0.0000,
    booking_rate DECIMAL(5, 4) NOT NULL DEFAULT 0.0000,
    
    -- 时间戳
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    
    UNIQUE KEY uq_vacancy_stats (hotel_id, room_type_id, stat_date, stat_hour)
);
```

**2. 实现定时任务**

```java
@Component

    @Scheduled(cron = "0 0 1 * * ?")  // 每天凌晨1点执行

        // 计算前一天的全天统计
```

**3. 混合查询策略**

```java
public List<VacancyPoint> buildPoints(request) {
    List<VacancyPoint> result = new ArrayList<>();
    
    // 1. 查询数据库预计算数据（历史）

        request.getHotelId(),
        request.getRoomTypeId(),
        request.getStartDate(),
        request.getEndDate().minusHours(1)  // 当前小时之前

    result.add(livePoint);
```

## 问题三：空置曲线警告逻辑错误

### 问题描述

在实现空置率阈值警告功能时，出现**误报**问题：

- 空置率刚好等于85%时，被标记为警告（应该只在**超过**85%时警告）
- 空置率刚好等于15%时，被标记为警告（应该只在**低于**15%时警告）

**错误代码**：
```java
// AnalyticsServiceImpl.java
private ThresholdAlert evaluateAlert(double rate, double high, double low) {
    if (rate >= high) {  // ❌ 错误：包含了等于的情况
        return new ThresholdAlert("HIGH", "超出上限");
    }
    if (rate <= low) {   // ❌ 错误：包含了等于的情况
        return new ThresholdAlert("LOW", "低于下限");
    }
    return null;
}
```

### 问题分析

- 使用`>=`和`<=`导致**边界值**被误判
- 阈值本身应该是**警戒线**，只有跨越才需要警告
- 数学上：警告区间应该是`(high, 1]`和`[0, low)`，而非`[high, 1]`和`[0, low]`

### 解决方案

修改比较运算符，改为严格不等式：

```java
private ThresholdAlert evaluateAlert(double rate, double high, double low) {
    if (rate > high) {   // ✅ 正确：仅当严格大于时警告
        return new ThresholdAlert("HIGH", "超出上限");
    }
    if (rate < low) {    // ✅ 正确：仅当严格小于时警告
        return new ThresholdAlert("LOW", "低于下限");
    }
    return null;
}
```

### 运行结果

✅ **问题已解决**

测试用例验证：

|--------|--------|--------|----------|----------|
| 0.90 | 0.85 | 0.15 | ⚠️ HIGH | ✅ HIGH |
| 0.85 | 0.85 | 0.15 | ✅ 无警告 | ✅ 无警告 |
| 0.50 | 0.85 | 0.15 | ✅ 无警告 | ✅ 无警告 |
| 0.15 | 0.85 | 0.15 | ✅ 无警告 | ✅ 无警告 |
| 0.10 | 0.85 | 0.15 | ⚠️ LOW | ✅ LOW |

## 问题五：预测算法单调趋向极值

### 问题描述

初始实现的预测算法过于简单，导致预测曲线**快速趋向100%或0%**：

```java
// ❌ 错误算法
double forecast = baseline + trend * i;  // 线性增长，无衰减
```

**表现**：
- 如果最近趋势是上升，预测7天后达到100%
- 如果最近趋势是下降，预测7天后降至0%
- 不符合真实业务场景（入住率会波动，不会持续单向）

### 问题分析

真实的空置率变化具有以下特点：
1. **周期性波动**：周末vs工作日、节假日vs平日
2. **均值回归**：长期会趋向历史平均值
3. **随机性**：受天气、活动等因素影响
4. **合理范围**：不会长期停留在极值（0%或100%）

### 解决方案

#### 新算法：多频率波动 + 趋势衰减 + 均值回归

```java
private List<VacancyPoint> forecastPoints(
    LocalDateTime lastTimestamp,
    double baseline,
    double historicalMean,
    int count
) {
    List<VacancyPoint> result = new ArrayList<>();
    
    for (int i = 1; i <= count; i++) {
        LocalDateTime ts = lastTimestamp.plusHours(i);
        
        // 1. 趋势衰减（避免极值）
        double trend = baseline - historicalMean;
        double decayedTrend = trend * Math.pow(0.95, i);
        
        // 2. 三频率波动叠加
        double fastOsc = 0.12 * Math.sin(i * 0.6);    // 快速波动（小时级）
        double midOsc = 0.08 * Math.sin(i * 0.3);     // 中速波动（半天级）
        double slowOsc = 0.05 * Math.sin(i * 0.15);   // 慢速波动（天级）
        
        // 3. 均值回归（长期趋势）
        double reversion = (historicalMean - baseline) * 0.02 * i;
        
        // 4. 合成最终预测值
        double forecast = baseline + decayedTrend + fastOsc + midOsc + slowOsc + reversion;
        
        // 5. 范围限制
        forecast = Math.max(0.10, Math.min(0.90, forecast));
        
        result.add(new VacancyPoint(ts, forecast, "FORECAST"));
    }
    
    return result;
}
```

**算法参数说明**：
- `Math.pow(0.95, i)`：趋势每小时衰减5%
- `0.12/0.08/0.05`：波动幅度（12%/8%/5%）
- `0.6/0.3/0.15`：波动频率（快/中/慢）
- `0.02 * i`：均值回归速度（每小时2%）
- `[0.10, 0.90]`：预测值限制在10%-90%区间

### 运行结果

✅ **问题已解决**

预测效果对比：

| 算法 | 7天后预测值 | 波动特性 | 真实性 |
|------|------------|----------|--------|
| 旧算法 | 98% | ❌ 单调上升 | ❌ 不合理 |
| 新算法 | 65% | ✅ 持续波动 | ✅ 符合现实 |

```java
    }
    
    return lines;
}
```
